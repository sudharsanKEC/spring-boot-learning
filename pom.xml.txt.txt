the pom.xml tailored for Java 21 (OpenJDK 21 LTS):
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
         http://maven.apache.org/xsd/maven-4.0.0.xsd">

    <modelVersion>4.0.0</modelVersion>

    <groupId>com.example</groupId>
    <artifactId>fullstack-backend-java</artifactId>
    <version>1.0-SNAPSHOT</version>

    <!-- Use Java 21 -->
    <properties>
        <maven.compiler.source>21</maven.compiler.source>
        <maven.compiler.target>21</maven.compiler.target>
        <maven.compiler.release>21</maven.compiler.release>
    </properties>

    <dependencies>
        <!-- JUnit for testing -->
        <dependency>
            <groupId>org.junit.jupiter</groupId>
            <artifactId>junit-jupiter</artifactId>
            <version>5.10.0</version>
            <scope>test</scope>
        </dependency>
    </dependencies>

    <build>
        <plugins>
            <!-- Surefire plugin for running tests -->
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-surefire-plugin</artifactId>
                <version>3.1.2</version>
            </plugin>
        </plugins>
    </build>

</project>

Notes for future reference:
	maven.compiler.source & maven.compiler.target → tells Maven which Java version to compile for.
	maven.compiler.release → ensures full Java 21 compatibility.
	JUnit 5 → standard for modern Java unit testing.
	Surefire plugin → needed to run tests with Maven.





1️⃣ What is pom.xml and why we need it
	pom.xml stands for Project Object Model in Maven.
	Its main roles are:
		Project Configuration & Metadata
			Defines project name, version, group, and packaging type.
			Lets other developers or build tools know what the project is.
		Dependency Management
			Declares external libraries your project needs (like JUnit, Spring, etc.).
			Maven downloads these automatically, so you don’t have to manually manage .jar files.
		Build Management
			Defines how your project is compiled, tested, packaged, and deployed.
			Can integrate plugins for extra features like code coverage, Docker builds, etc.
		Consistency
			Standardizes Java projects across teams.
			No matter which machine or IDE, Maven knows exactly how to compile, run, and test your project.

2️⃣ What if there was no pom.xml?
	If there is no pom.xml:
		You cannot use Maven for build automation.
		You would have to:
			Download every required .jar file manually.
			Set classpaths manually when compiling.
			Write scripts for compiling, running, testing.
			Keep track of library versions yourself.
		Problem: Tedious, error-prone, especially for large projects.

	Historical Context
		Before Maven (and even Gradle later):
			Developers manually wrote scripts or used Ant.
			Every project maintained its lib/ folder with jar files, manually included in classpath.
			Updating libraries required manual download.
			Build scripts were often different for each project → inconsistency.
		Maven solved these problems by standardizing the project structure, automating dependency management, and centralizing builds.


3️⃣ Explain the pom.xml
    Let’s go line by line.
    <project xmlns="http://maven.apache.org/POM/4.0.0"
            xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
            xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
            http://maven.apache.org/xsd/maven-4.0.0.xsd">

        Declares this file as a Maven project (project tag).
        xmlns → XML namespace for Maven POM version 4.0.0.
        xsi:schemaLocation → tells tools where to find the XML schema to validate the file.

    <modelVersion>4.0.0</modelVersion>
    Indicates POM model version (always 4.0.0 for modern Maven).

    <groupId>com.example</groupId>
    <artifactId>fullstack-backend-java</artifactId>
    <version>1.0-SNAPSHOT</version>
        groupId → like your company or organization (com.example).
        artifactId → your project name (fullstack-backend-java).
        version → project version (1.0-SNAPSHOT means development version).
    
    Java Version Configuration
        <properties>
            <maven.compiler.source>21</maven.compiler.source>
            <maven.compiler.target>21</maven.compiler.target>
            <maven.compiler.release>21</maven.compiler.release>
        </properties>

        source → which Java syntax to allow (Java 21 features like records, pattern matching, etc.).
        target → JVM version for compiled bytecode.
        release → ensures compiled bytecode matches JDK 21 APIs exactly.
        Using all three is common for modern Java projects to ensure consistency.

    Dependencies
        <dependencies>
            <dependency>
                <groupId>org.junit.jupiter</groupId>
                <artifactId>junit-jupiter</artifactId>
                <version>5.10.0</version>
                <scope>test</scope>
            </dependency>
        </dependencies>
        
        dependencies → list of external libraries.
        JUnit 5 (junit-jupiter) is used for testing.
        scope = test → used only during testing, not included in final build.
        Maven automatically downloads this from Maven Central.


    Build Plugins
        <build>
            <plugins>
                <plugin>
                    <groupId>org.apache.maven.plugins</groupId>
                    <artifactId>maven-surefire-plugin</artifactId>
                    <version>3.1.2</version>
                </plugin>
            </plugins>
        </build>

        Plugins extend Maven functionality.
        maven-surefire-plugin → runs your unit tests when you run mvn test.
        Without this, mvn test might fail or not run JUnit properly.

Summary Notes
    pom.xml = project brain: knows dependencies, build, versioning.
    Without it → manual dependency management, inconsistent builds, hard to maintain.
    Maven solves automation, consistency, reproducibility.
    Modern Java projects always use Maven/Gradle for large, maintainable projects.


Did developers write pom.xml manually in olden days?
        Yes, Maven has been around since the early 2000s.
        Developers did write pom.xml manually, because Maven needed project metadata, dependencies, and build instructions.
        Early on, you could use IDE tools (like Eclipse Maven plugin) to generate a basic pom.xml, but the language itself is XML, so yes, it’s all written in XML syntax.
    ✅ Notes:
        XML = eXtensible Markup Language, like HTML, but for data representation.
        It’s structured and human-readable, but verbose.
        Every <tag> has a meaning that Maven understands.


What is a JAR file?
    JAR = Java ARchive
        It’s basically a compressed file (like .zip) that contains compiled Java classes (.class) and resources.
        Think of it as a ready-to-use library for Java projects.
    For example:
        You write some utility functions in Java, compile them, then package into utils.jar.
        Other projects can just import utils.jar instead of copying individual .java files.
    ✅ Notes:
        JAR files can also include:
            Metadata (MANIFEST.MF)
            Configuration files (like .properties)
        Your own project can also be packaged as a JAR if it’s a library.


How Maven works with JAR files
    Maven:
        Reads your pom.xml.
        Finds all dependencies (e.g., JUnit, Spring) listed in <dependencies>.
        Downloads the required JARs from Maven Central repository.
        Puts them in a local cache (.m2 folder on your computer).
        Automatically sets the classpath for compilation and execution.
    ✅ Notes:
        This means you never have to manually download jar files or set classpaths.
        Maven manages versions, conflicts, and transitive dependencies.

Are JAR files similar to Node modules in Express?
    Yes, conceptually very similar:
        | Java (Maven)                              | Node.js (npm/Express)                      |
        | ----------------------------------------- | ------------------------------------------ |
        | `.jar` files                              | `node_modules` folder                      |
        | Maven downloads JARs and manages versions | npm installs packages and manages versions |
        | `pom.xml` lists dependencies              | `package.json` lists dependencies          |
        | Automatic classpath setup                 | `require()` or `import` loads modules      |

    Difference:
        In Java, the .jar is binary compiled code.
        In Node, the modules are usually source JS code, interpreted at runtime.
    Conceptually, both solve the same problem: dependency management.

Example Flow (Old way vs Maven)
    Old way (pre-Maven):
        1) Download junit-5.10.0.jar.
        2) Place in lib/ folder.
        3) Add -cp lib/junit-5.10.0.jar to compile command:
             javac -cp lib/junit-5.10.0.jar src/main/java/com/example/App.java
        4) Manually update if JAR version changes.

Maven needs an internet connection the first time to download dependencies from the central repository. Once downloaded, they are cached locally in the .m2 folder, so later builds don’t require internet.
